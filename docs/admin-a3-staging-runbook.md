# A3 staging handoff

This draft does not authorize a deployment, migration, customer email, or payment.

1. Review the additive migration `20260720110000_admin_operational_actions.sql`, including its rollback comment, then apply it only to the isolated Staging Supabase project.
2. Use newly created, removable `A3-STAGING-*` fixture rows and an approved manager UUID. Do not use customer or Production data.
3. Verify RLS remains enabled and browser clients have no mutation policies for the new tables.
4. Confirm a manager can approve a manual quote and an operator cannot; confirm unauthenticated users cannot invoke an action.
5. Confirm invitation send creates or reuses one draft invitation, stores only a SHA-256 token hash, gives the attempt a seven-day expiry, and records a **suppressed** email outcome without changing invitation or reservation sent state. A confirmed injected test adapter may advance the invitation and reservation; suppressed and failed outcomes must not. A new retry key creates a new attempt, while the same key replays one result. Confirm finalization preserves forward `order_started`, payment and converted lifecycle states.
6. Confirm resend rotates the active token only after confirmed delivery. Verify token hashes and plaintext tokens are absent from Admin reads, API responses, audit details, entity events and final email history. The internal attempt table and token-bearing delivery payload RPC are server-only and must not receive a browser read policy. Rotate the local token secret during a pending-attempt test and confirm the attempt fails closed without dispatching a dead link.
7. Exercise two concurrent handlers for one attempt: the five-minute database claim must allow one dispatcher only, the other must receive `operation_in_progress`, and completion must create one delivery event. After the lease expires, confirm another active operator can reclaim the same attempt and provider idempotency key. The real provider adapter must honor that key before email delivery can be enabled.
8. Confirm manual quotes are manager-only, future-dated, restricted to server-classified manual-review destinations and atomically replaced. Confirm approval bumps the locked invitation timestamp so two confirmations from one preview conflict. Checkout stores `manual_shipping_quote_id` and the order trigger rejects a cancelled, expired, mismatched-invitation/country/currency/amount quote on every dependent order update before payment creation.
9. Confirm a Mollie `payments` row with `status = paid`, provider ID, webhook timestamp, paid timestamp, matching amount and matching currency is required before every fulfilment transition. Exercise `unfulfilled -> ready_to_pack -> packed -> shipped`, stale version conflicts, whitespace/length validation, and the shipping-confirmation suppressed/failed/sent histories. For a suppressed or failed shipping confirmation, use the contextual `shipping.retry` preview and confirmation; it must not repeat the fulfilment transition.
10. Verify each finalized operational email writes exactly one correlated email event, admin audit event and entity event in the same transaction. Confirm invitation and reservation lifecycle changes add their own before/after events without PII or token data. Verify email history is append-only and does not expose recipients, provider IDs, message bodies, token hashes or secrets through Admin reads.
11. The legacy `x-admin-action-secret` invitation sender is intentionally a `410 Gone` compatibility tombstone; it cannot send or mutate. Only the authenticated Admin action contract is active.
12. Migration/code rollout order matters: apply the reviewed A3 migration before enabling manual international quotes. Automatic NL/EU quote and checkout paths do not query the A3 quote table and remain compatible before migration.
13. No local PostgreSQL/Supabase runtime is configured in this repository. Node tests validate production handlers and the SQL contract structure, but real concurrent-session, trigger and migration execution must still be proven in the isolated Staging project before any release approval.
14. Rollback requires revoking/dropping the six A3 RPCs and order validation trigger/function before removing the A3 tables, indexes, constraints and additive order columns. Preserve required audit exports first; do not roll back a quote referenced by an order.
15. Do not create Mollie payments, write WooCommerce, deploy, merge, or access Production as part of this handoff.
